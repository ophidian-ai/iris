Architecture Approach

Three ways to build the overhauled social content engine

A

Monolithic Skill Overhaul

Rewrite the existing social-content SKILL.md as one big skill that handles everything: content generation, image pipeline routing, platform API integration, scheduling, and approval workflow. Single entry point, single file.


Pros

  • Simple to invoke -- one command does everything
  • All logic in one place
  • Fastest to build initially

Cons

  • Massive skill file -- hard to maintain
  • Can't reuse image pipeline or scheduling for client-social-content
  • Testing individual pieces is difficult
  • Doesn't scale to client offering
B

Modular Pipeline (Recommended)

Break into composable sub-skills that chain together. The social-content skill becomes the orchestrator that calls: (1) content generator, (2) image pipeline, (3) platform scheduler. Each piece works independently and can be reused by client-social-content.


Pros

  • Reusable -- client skill uses the same image pipeline and scheduler
  • Testable -- each module works independently
  • Maintainable -- change scheduling without touching content generation
  • Scales to client offering naturally

Cons

  • More files to manage
  • Slightly more complex to build initially
  • Need to define interfaces between modules
C

Script-Driven Engine

Build the pipeline as Node.js scripts (like the GWS CLI scripts) rather than skill files. The skill just orchestrates the scripts. Image generation, API calls, and scheduling are all executable scripts in a scripts/ folder.


Pros

  • Scripts can run on cron/schedule without Claude
  • Most portable -- works outside the skill system
  • Easiest to debug (run scripts individually)

Cons

  • Content generation still needs Claude (can't script AI writing)
  • Two systems to maintain (scripts + skill)
  • Loses the "one command" simplicity for the AI-driven parts
